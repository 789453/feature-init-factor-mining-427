# 2. AlphaPROBE 哪些部分值得借鉴，哪些不建议直接复用

我重新看了 AlphaPROBE 的公开实现与论文说明。AlphaPROBE 官方 README 和论文都强调其核心是 **Bayesian Factor Retriever + DAG-aware Factor Generator**，也就是把因子池建成 DAG，在图上做检索和演化。这个思想适合后期因子库管理和二阶段优化，但不适合作为你第一阶段的主搜索引擎。([GitHub](https://github.com/gta0804/AlphaPROBE?utm_source=chatgpt.com "Official implementation of AlphaPROBE: Alpha mining with ... - GitHub"))

## 可以借鉴的部分

### 1. `calculator.py`

AlphaPROBE 的 `AlphaCalculator` 抽象接口把单因子 IC、RankIC、因子间 IC、因子组合 IC 都抽象出来。这个非常值得借鉴，因为它把“表达式生成”和“因子评估”解耦。公开代码里接口包含 `calc_single_IC_ret`、`calc_single_rIC_ret`、`calc_mutual_IC`、`calc_pool_IC_ret` 等方法。([GitHub](https://raw.githubusercontent.com/gta0804/AlphaPROBE/master/src/alphagen/data/calculator.py "raw.githubusercontent.com"))

我在 MVP 中对应实现的是：

```
metrics.py
evaluator.py
pipeline.py

```

现在还不是严格抽象类，但逻辑已经解耦：表达式生成不直接知道 IC 怎么算，evaluator 只接收表达式和数据面板。

### 2. `expression.py`

AlphaPROBE 的 `Expression` / `Feature` / `Constant` / `UnaryOperator` / `BinaryOperator` / `RollingOperator` / `PairRollingOperator` 设计很有参考价值。它把算子按类别划分，rolling 算子通过 `unfold` 做窗口计算，pair rolling 算子单独处理 `TsCov / TsCorr`。([GitHub](https://raw.githubusercontent.com/gta0804/AlphaPROBE/master/src/alphagen/data/expression.py "raw.githubusercontent.com"))

这次 MVP 也采用了类似分类：

```
UNARY
BINARY
ROLLING
PAIR_ROLLING

```

但没有直接硬拷贝 AlphaPROBE 的 `FeatureType`，因为你的字段不会硬编码，未来字段应该动态来自配置和数据表。

### 3. `tree.py`

AlphaPROBE 的 parser 支持从函数式表达式转 RPN / tree，并且有 `ExpressionBuilder.validate()` 来检查 token 是否能构造合法树。公开代码里 `ExpressionParser` 的 feature map 仍然硬编码 `$open/$close/$high/$low/$volume/$vwap`，这一点不适合你的多源字段体系。([GitHub](https://raw.githubusercontent.com/gta0804/AlphaPROBE/master/src/alphagen/data/tree.py "raw.githubusercontent.com"))

MVP 里我做成了：

```
$field_name 动态解析
parse_expr("Rank(TsMean($ret_1d,10))")

```

字段名来自 `DEFAULT_FEATURES`，后面可以来自 YAML。

### 4. `dimension.py`

AlphaPROBE 的 `dimension.py` 做了表达式维度计算，判断不同算子的维度变化。例如 `Rank` 返回无量纲，`Div` 做维度相减，`Mul` 做维度相加。([GitHub](https://raw.githubusercontent.com/gta0804/AlphaPROBE/master/src/alphagen/data/dimension.py "raw.githubusercontent.com"))

这部分很适合你后续增强，但第一版我没有做复杂 dimension algebra，而是先做：

```
字段合法
算子合法
窗口合法
arity 合法
复杂度合法
Log / Pow 数值风险合法

```

原因是：你目前字段本身已经是 domain feature，不是 raw price/volume。第一阶段先用数值稳定性和结构复杂度控制更重要。

### 5. `expression_knowledge_graph.py`

AlphaPROBE 的知识图谱代码把表达式包装成 node，记录 token、长度、topic、description、IC、ICIR、parent、children 等。这个非常适合作为你后续“因子库 + 模板库 + 表达式谱系”的基础思想。([GitHub](https://raw.githubusercontent.com/gta0804/AlphaPROBE/master/src/alphagen/data/expression_knowledge_graph.py "raw.githubusercontent.com"))

但第一阶段不要上 DAG-aware evolution。先把每个表达式的：

```
expr
canonical
status
IC
RankIC
ICIR
coverage
turnover
yearly IC

```

稳定记录下来即可。

# 3. Typed grammar 和模板枚举是什么关系

你可以这样理解：

```
Typed grammar = 语法边界和类型规则
模板枚举 = 在合法边界内高效生成候选表达式

```

它们不是替代关系，而是上下游关系。

## Typed grammar 负责回答：

```
这个表达式能不能存在？
这个算子能不能接这个输入？
这个窗口是否合法？
这个表达式是不是太深？
这个表达式有没有明显数值风险？
这个字段是否允许进入这个算子？

```

例如：

```
TsCorr($main_net_ratio,$ret_1d,20)

```

合法。

但：

```
TsCorr($main_net_ratio,10,$ret_1d)

```

非法，因为窗口参数位置错。

又比如：

```
Log($ret_1d)

```

第一阶段默认非法，因为收益率可能为负。

## 模板枚举负责回答：

```
在合法表达式空间里，哪些结构最值得优先测试？

```

例如优先枚举：

```
Rank(TsMean($x,20))
Rank(TsDelta($x,20))
Rank(TsCorr($x,$y,20))
Sub(Rank(TsMean($x,20)),Rank(TsMean($y,20)))

```

而不是无脑枚举：

```
Pow(Mul(Sign(Greater(...)),...),...)

```

## 这次 MVP 中的关系

项目里：

```
grammar.py 负责模板枚举
validator.py 负责 typed grammar / 合法性校验
parser.py 负责表达式解析和 canonicalization

```

这样解耦之后，后面你可以让 Ollama / LLM / GP / 手工模板都只输出字符串表达式，然后统一交给 validator。

# 5. 算子选择建议

你给的算子列表：

```
Abs, SLog1p, Inv, Sign, Log, Rank,
Add, Sub, Mul, Div, Pow, Greater, Less,
Ref, TsMean, TsSum, TsStd, TsIr, TsMinMaxDiff, TsMaxDiff, TsMinDiff,
TsVar, TsSkew, TsKurt, TsMax, TsMin, TsMed, TsMad, TsRank, TsDelta,
TsDiv, TsPctChange, TsWMA, TsEMA,
TsCov, TsCorr

```

我在 MVP 中基本都实现了，但模板枚举阶段没有把所有算子都作为主力。

## 第一阶段主力算子

```
Rank
SLog1p
TsMean
TsStd
TsIr
TsRank
TsDelta
TsDiv
TsPctChange
TsWMA
TsEMA
TsCorr
Sub
Div

```

这些适合短期资金/量价因子。

## 保留但不建议主力使用

```
Pow
Sign
Greater
Less
Mul
TsSkew
TsKurt
TsCov

```

原因：

算子

判断

`Pow`

容易制造极端值和过拟合，只允许常数 0.5/2/3

`Sign`

信息损失大，适合后期离散信号，不适合初筛主力

`Greater/Less`

更像 winsor/clip 或 max/min，不适合大规模自由枚举

`Mul`

量纲和极端值风险高，除非两个输入都已 rank/zscore

`TsSkew/TsKurt`

对窗口和异常值高度敏感，初筛可先低优先级

`TsCov`

量纲敏感，通常不如 `TsCorr` 稳健

# 6. 项目中已经实现的模板

当前 `grammar.py` 包含四类模板。

## 模板 1：单字段时序变换 + 截面 Rank

```
Rank(TsMean($x,w))
Rank(TsStd($x,w))
Rank(TsIr($x,w))
Rank(TsRank($x,w))
Rank(TsDelta($x,w))
Rank(TsPctChange($x,w))
Rank(TsWMA($x,w))
Rank(TsEMA($x,w))

```

这是第一阶段主力，覆盖趋势、波动、变化率、平滑、排序结构。

## 模板 2：两字段结构差

```
Sub(Rank(TsMean($x,w)),Rank(TsMean($y,w)))
Sub(Rank(TsDelta($x,w)),Rank(TsDelta($y,w)))
Div(Rank(TsMean($x,w)),Rank(TsMean($y,w)))

```

适合：

```
大单 vs 小单
资金流 vs 换手
筹码压力 vs 获利盘
量价强度 vs 振幅

```

## 模板 3：Pair rolling 相关性

```
Rank(TsCorr($x,$y,w))

```

适合测试：

```
资金流和收益是否同步
筹码位置和换手是否共振
大单净流和短期价格是否背离

```

## 模板 4：人工方向性组合

例如：

```
Rank(TsCorr($big_vs_small_flow,$ret_1d,20))
Sub(Rank(TsMean($main_net_ratio,20)),Rank(TsMean($turnover_log,20)))
Rank(TsDelta($active_big_buy_pressure,20))

```

这类模板比纯枚举更有经济含义。

# 9. 当前项目和未来高性能版本的区别

现在 MVP 重点是**跑通工程闭环**，不是一步到位做极限性能。

当前：

```
Pandas rolling + NumPy panel

```

后续建议：

```
DuckDB/Polars 做数据读取和过滤
NumPy 做 T x N 面板
Numba 加速 rolling rank / rolling corr
表达式 DAG 做公共子表达式缓存

```

Polars 的 `scan_parquet` 是 lazy scan，会延迟读取并支持优化器、谓词下推、投影下推，这适合后续从 silver/gold Parquet 做高性能筛选。([docs.pola.rs](https://docs.pola.rs/user-guide/io/parquet/?utm_source=chatgpt.com "Parquet - Polars user guide - GitHub Pages"))

Numba 的 `vectorize` 可以把纯 Python 标量函数编译成 NumPy ufunc 风格执行，适合后续把部分数值算子和循环核加速。([numba.pydata.org](https://numba.pydata.org/numba-doc/dev/user/vectorize.html?utm_source=chatgpt.com "Creating NumPy universal functions — Numba 0.52.0.dev0+274 ... - PyData"))
