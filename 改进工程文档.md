# 二阶段工程文档：以“统一模板生成 + 有效性统计反馈 + 可断点去重挖掘”为核心重构

这版我会按你的反馈重新定义二阶段目标：**不再按字段类别定制模板**，而是把所有字段视为同一类短期量价/资金/筹码衍生特征，统一进入同一套模板生成器；股票池只承担筛选效率任务，不把它设计得过于复杂；真正的核心放在两件事上：第一，扩展模板空间，让三阶、四阶、多维组合、一阶变换、时序变换、截面变换都能系统生成；第二，建立表达式—字段—算子—模板族—窗口—效果的统计反馈闭环，用后续实际挖掘结果反推哪些模板、算子、字段、窗口组合有效。

当前仓库本身已经是一个适合继续演进的原型：README 明确写了第一阶段 MVP 的能力，包括从 DuckDB 或模拟数据加载 daily / moneyflow / cyq\_perf / daily\_basic，构建约 30 个短期基础特征字段，使用 typed grammar + 模板枚举生成低阶表达式，做解析、合法性校验、复杂度控制、表达式去重，在线计算表达式，批量计算 IC / RankIC / ICIR / 覆盖率 / 换手 proxy / 年度和分段稳定性，并记录表达式测试结果，同时预留二阶段优化接口。这个基础不需要推翻，但要把 `grammar.py`、`metrics.py`、`store.py`、`pipeline.py` 四个核心模块升级成“可扩展模板挖掘系统”。([GitHub](https://github.com/789453/feature-init-factor-mining-427/tree/fast-update-427 "GitHub - 789453/feature-init-factor-mining-427 at fast-update-427 · GitHub"))

## 0. 先修正几个设计原则

### 0.1 股票池：二阶段用于筛选，不承担最终真实性验证

你说得对，二阶段股票池的目的不是证明最终策略一定有效，而是**在计算量可控的条件下快速筛表达式**。如果已有股票池已经剔除 ST，并按流动性、市值、行业做了分桶分层抽样，那么二阶段可以采用“年度重抽样 + 固定随机种子 + 少量扰动”的方式，而不是我之前建议的过细动态 universe。

建议实现为：

```
PoolMode:
  static_pool: 直接使用 pool_json
  annual_resample: 每年按已有分桶逻辑重抽一次
  annual_resample_randomized: 每年重抽，并允许 seed 控制随机性
  full_universe_final: 只用于最后全量复核

```

实际流程是：

```
2018-2026 全量数据
→ 每年抽 1000 只左右股票作为 coarse pool
→ 用 coarse pool 跑全模板粗筛
→ 选 top-K + 概率抽样尾部候选进入 fine pool
→ 最后只对候选在更大池或全量池显示结果

```

这里的重点是：股票池不要变成主要研究对象。二阶段的主要研究对象应该是“模板是否足够丰富”和“统计反馈是否能闭环”。

***

### 0.2 字段：全部字段统一使用同一套模板

我同意你的修正。当前字段虽然来源上有价格、量、资金、筹码、流动性之分，但在挖掘层面，它们都已经被 `fields.py` 处理成短期可比的衍生字段。代码里 `add_basic_features` 统一构造了 `ret_1d`、`hl_range`、`oc_ret`、`close_pos`、`gap_ret`、`intraday_reversal`、`amount_log`、`vol_log`、`turnover_log`、`price_volume_pressure`、`amplitude_turnover`、`volume_ratio_x_ret` 等字段，随后又构造了资金流、筹码、胜率、历史位置等字段，最后 `DEFAULT_FEATURES` 统一列出 30 个默认特征。也就是说，从工程视角看，它们已经是同一个 feature namespace，而不是多个独立领域。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/fields.py "feature-init-factor-mining-427/src/alpha_mvp/fields.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

因此二阶段模板生成必须遵循：

```
for field in selected_fields:
    apply same unary transforms
    apply same time-series transforms
    apply same cross-sectional transforms
    apply same pair/triple/multi-field composition templates

```

字段类别可以保留，但只用于**统计分析和结果解释**，不用于生成差异化模板。比如 `field_group = price_like / flow_like / chip_like / liquidity_like` 可以作为 `expression_meta` 里的标签，后续分析时看哪个 group 整体贡献更强；但模板生成器不应该因为某字段属于筹码类就只给它筹码模板，也不应该因为某字段属于量价类就给它更多模板。这样后续字段增加、删除、替换时，模板体系不会被字段语义绑定死。

### 0.3 模板：二阶段的第一主线

这次需要直切核心：**模板空间不够丰富**是目前最大瓶颈。当前 `grammar.py` 的生成方式比较简单：先定义 `CORE_TS = ["TsMean", "TsStd", "TsIr", "TsMinMaxDiff", "TsRank", "TsDelta", "TsDiv", "TsPctChange", "TsWMA", "TsEMA"]`，然后生成四类模板：单字段模板、二字段 Sub/Div 模板、pair corr 模板、少量 hard-coded directional pairs。单字段模板主要是 `Rank(TsOp($f,w))`，部分时序算子再包一层 `SLog1p`；二字段模板主要是 `Sub(Rank(TsMean(f1)), Rank(TsMean(f2)))`、`Sub(Rank(TsDelta(f1)), Rank(TsDelta(f2)))`、`Div(Rank(TsMean(f1)), Rank(TsMean(f2)))`；pair corr 则只对前 18 个字段做 `Rank(TsCorr(f1,f2,w))`。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/grammar.py "feature-init-factor-mining-427/src/alpha_mvp/grammar.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

这个设计适合作为 MVP，但二阶段应该把模板从“手写几类公式”升级为“受约束的 typed expression grammar”。也就是说，模板不是列举几行字符串，而是由若干层算子族组合生成：

```
Field layer:
  F = $field

Unary layer:
  U(F) = Id(F), Abs(F), SLog1p(F), Sign(F), Inv(F), LogSafe(F), Pow2(F), SqrtAbs(F)

Time-series layer:
  T(U(F), w) = TsMean, TsStd, TsIr, TsRank, TsDelta, TsDiv,
               TsPctChange, TsMinMaxDiff, TsWMA, TsEMA,
               TsMaxDiff, TsMinDiff, TsMad, TsSkew, TsKurt

Cross-section layer:
  X(T) = Id(T), Rank(T), SLog1p(T), Rank(SLog1p(T)), SLog1p(Rank(T))

Binary composition:
  B(A, B) = Add, Sub, Mul, Div, Greater, Less

Pair rolling:
  P(A, B, w) = TsCorr(A, B, w), TsCov(A, B, w)

Higher-order:
  H3 = B(B(A, B), C), B(A, B(B, C))
  H4 = B(B(A, B), B(C, D))

```

这个结构才能真正产生更丰富的三阶、四阶、多维组合，同时仍然受 validator 控制。

## 2. 模板系统重构：从字符串枚举变成 TemplateSpec

### 2.1 为什么当前 `grammar.py` 不够

当前 `grammar.py` 的优点是简单、清晰、容易调试；缺点是生成逻辑写死在函数里，模板族不够细，表达式扩展只能不断在 `generate_all_templates` 中追加字符串。比如现在的二字段结构主要只有 `Sub` 和 `Div`，这正是你指出的问题：组合结构应该不止减法和除法，还应该有 `Add`、`Mul`、`Greater`、`Less`、平滑差、短长窗口差、双字段乘积、嵌套组合、三字段组合、四字段组合等。当前 `grammar.py` 虽然有 `allow_heavy_ops`，但 heavy 主要影响 `TsCov`，并没有真正扩展组合模板。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/grammar.py "feature-init-factor-mining-427/src/alpha_mvp/grammar.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

二阶段应该把模板生成拆成三层：

```
1. primitive expression 生成
2. composition expression 组合
3. validation + canonical 去重 + meta 记录

```

当前 `parser.py` 已经提供了 `Node`、`parse_expr` 和 `canonical`，并且 canonical 对 `Add`、`Mul`、`TsCorr`、`TsCov` 这类交换律算子会排序参数，正好可以继续用于去重。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/parser.py "feature-init-factor-mining-427/src/alpha_mvp/parser.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

```
    order: int
    enabled: bool = True

    short_windows: Sequence[int] = ()
    long_windows: Sequence[int] = ()
```

然后把模板写成配置：

### 2.3 一阶模板：不是只有 `Rank(TsOp(f,w))`

一阶模板要变成“字段 → 一阶变换 → 时序变换 → 截面变换”的笛卡尔结构，但要有约束，否则数量会膨胀。当前 `ops.py` 已有 `Abs`、`SLog1p`、`Inv`、`Sign`、`Log`、`Rank` 等一元算子，也已有 `Add`、`Sub`、`Mul`、`Div`、`Pow`、`Greater`、`Less` 等二元算子。时序算子方面，当前已实现 `TsMean`、`TsSum`、`TsStd`、`TsVar`、`TsMax`、`TsMin`、`TsMed`、`TsDelta`、`TsPctChange`、`TsDiv`、`TsIr`、`TsMinMaxDiff` 等。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/ops.py "feature-init-factor-mining-427/src/alpha_mvp/ops.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

此外，`ops.py` 还实现了 `TsRank`、`TsWMA`、`TsEMA`、`TsCov`、`TsCorr` 等操作，其中 `TsRank`、`TsWMA`、`TsEMA` 对短周期量价特征很重要，因为它们分别对应时序位置、线性衰减和平滑记忆。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/ops.py "feature-init-factor-mining-427/src/alpha_mvp/ops.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

一阶模板建议包括：

```
A1:
  Rank(TsOp($f,w))

A2:
  SLog1p(TsOp($f,w))

A3:
  Rank(SLog1p(TsOp($f,w)))

A4:
  Rank(TsOp(SLog1p($f),w))

A5:
  Rank(TsOp(Abs($f),w))

A6:
  Rank(TsOp(Sign($f),w))

A7:
  Rank(TsDelta(TsMean($f,w1),w2))    # 当前 validator 可能不允许，需要新限制

```

但 A7 这种嵌套时序会增加节点和 ts\_ops，因此不建议第一批放开太多。更好的办法是新增更原子的新算子，例如：

```
TsZScore($f,w) = Div(Sub($f,TsMean($f,w)),TsStd($f,w))
TsMeanDelta($f,w) = Sub($f,TsMean($f,w))
TsEmaDelta($f,w) = Sub($f,TsEMA($f,w))
TsStdRatio($f,w1,w2) = Div(TsStd($f,w1),TsStd($f,w2))
TsMeanRatio($f,w1,w2) = Div(TsMean($f,w1),TsMean($f,w2))

```

为什么建议新增算子而不是用表达式嵌套硬拼？因为当前 validator 默认最大深度 4、最大节点 10、最多 2 个时序算子、最多 1 个 pair 算子、最多 1 个 binary 算子；复杂模板如果都用现有算子拼，会很快超过上限。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/validator.py "feature-init-factor-mining-427/src/alpha_mvp/validator.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

### 2.4 二阶模板：从 Sub/Div 扩展到全组合结构

当前二阶模板主要是：

```
Sub(Rank(TsMean(f1,w)), Rank(TsMean(f2,w)))
Sub(Rank(TsDelta(f1,w)), Rank(TsDelta(f2,w)))
Div(Rank(TsMean(f1,w)), Rank(TsMean(f2,w)))

```

二阶段至少应扩展为以下结构：

```
B1:
  Rank(B(TsOp($f1,w), TsOp($f2,w)))

B2:
  Rank(B(Rank(TsOp($f1,w)), Rank(TsOp($f2,w))))

B3:
  SLog1p(B(Rank(TsOp($f1,w)), Rank(TsOp($f2,w))))

B4:
  Rank(B(TsOp1($f1,w), TsOp2($f2,w)))

B5:
  Rank(B(SLog1p(TsOp($f1,w)), SLog1p(TsOp($f2,w))))

B6:
  Rank(TsCorr($f1,$f2,w))

B7:
  SLog1p(TsCorr($f1,$f2,w))

```

其中 `B` 不应只限 `Sub/Div`，建议第一阶段扩到：

```
BINARY_CORE = ["Add", "Sub", "Mul", "Div"]
BINARY_EXT = ["Greater", "Less"]

```

不过要注意当前 `Greater` 和 `Less` 在 `ops.py` 中是 `np.maximum` 和 `np.minimum`，不是布尔条件门控；因此它们的语义是“取较大/较小值”，而不是 `where`。如果后续想做真正的门控模板，应该新增 `WherePos(x, y)`、`GatePos(x, y)`、`Clip(x, low, high)` 等专门算子，而不要误用 `Greater`。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/ops.py "feature-init-factor-mining-427/src/alpha_mvp/ops.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

推荐二阶模板第一批：

```
Rank(Add(A,B))
Rank(Sub(A,B))
Rank(Mul(A,B))
Rank(Div(A,B))
Rank(Greater(A,B))
Rank(Less(A,B))
SLog1p(Sub(A,B))
SLog1p(Mul(A,B))
Rank(TsCorr(A_raw,B_raw,w))

```

其中：

```
A = Rank(TsOp($f1,w)) or TsOp($f1,w)
B = Rank(TsOp($f2,w)) or TsOp($f2,w)

```

为了控制数量，二阶模板不要对所有 `TsOp1 × TsOp2` 全展开。可以分三档：

```
Tier 1:
  same_ts: TsMean/TsDelta/TsEMA/TsWMA/TsStd

Tier 2:
  selected_mixed_ts:
    TsMean × TsStd
    TsMean × TsDelta
    TsEMA × TsDelta
    TsIr × TsMean
    TsRank × TsMean

Tier 3:
  pair rolling:
    TsCorr/TsCov

```

### 2.5 三阶模板：核心是“两个字段形成状态，第三个字段调制”

三阶模板是二阶段最值得新增的部分。它不是为了追求复杂，而是因为很多短周期因子天然是“三个维度”的交互，例如：

```
价格状态 × 成交活跃度 × 资金流
振幅状态 × 位置状态 × 成交状态
反转状态 × 波动状态 × 流动性状态

```

但按照你的要求，模板生成时不按字段类别区分，因此工程上应该是对任意字段三元组统一应用：

```
C1:
  Rank(Mul(Sub(A,B), C))

C2:
  Rank(Div(Sub(A,B), Abs(C)))

C3:
  Rank(Sub(Mul(A,B), C))

C4:
  Rank(Add(Mul(A,B), C))

C5:
  Rank(Mul(Div(A,B), C))

C6:
  Rank(Sub(A, Div(B,C)))

C7:
  Rank(Div(A, Add(Abs(B), Abs(C))))

```

这些模板的形态可以理解为：

```
差值被第三变量调制
比值被第三变量调制
两个变量乘积再与第三变量比较
一个变量相对于两个变量组合的偏离

```

示例表达式：

```
Rank(Mul(Sub(Rank(TsMean($f1,20)),Rank(TsMean($f2,20))),Rank(TsStd($f3,20))))

Rank(Div(Sub(Rank(TsEMA($f1,20)),Rank(TsEMA($f2,20))),SLog1p(Abs(TsMean($f3,20)))))

Rank(Sub(Mul(Rank(TsDelta($f1,10)),Rank(TsMean($f2,10))),Rank(TsMean($f3,10))))

```

三阶组合会快速爆炸。如果字段 30 个，三元组合 `C(30,3)=4060`，每个组合再乘窗口、算子、二元结构，会非常大。因此三阶模板必须通过 `template_budget` 控制：

```
max_expr_per_template_family = 20000
field_tuple_sampling = deterministic hash sampling
operator_tuple_sampling = white list
window_tuple_sampling = same_window first, mixed_window second

```

***

### 2.6 四阶模板：只做平衡结构，不做任意嵌套

四阶模板可以有，但必须少量、结构化。不要生成任意树，因为四阶树容易过拟合，也容易超过 validator 限制。推荐只生成平衡结构：

```
D1:
  Rank(Sub(Mul(A,B), Mul(C,D)))

D2:
  Rank(Div(Mul(A,B), Add(Abs(C),Abs(D))))

D3:
  Rank(Sub(Div(A,B), Div(C,D)))

D4:
  Rank(Mul(Sub(A,B), Sub(C,D)))

D5:
  Rank(Add(Sub(A,B), Sub(C,D)))

```

示例：

```
Rank(Sub(Mul(Rank(TsMean($f1,20)),Rank(TsMean($f2,20))),
         Mul(Rank(TsMean($f3,20)),Rank(TsMean($f4,20)))))

```

四阶模板建议只在粗筛后作为二阶段增强使用，不应第一批全量展开。可以先让一阶、二阶、三阶完成主筛，四阶只对字段统计中表现较好的 top field subset 做，但注意“模板规则对所有字段一致”，这里的区别不是字段类别，而是**由统计结果产生的字段子集**。

```
def extract_expr_meta(node: Node, template_info=None) -> ExprMeta:
    fields = []
    ops = []
    windows = []

    def walk(n):
        if n.kind == "field":
            fields.append(n.value)
        elif n.kind == "op":
            ops.append(n.value)
            for a in n.args:
                if a.kind == "const":

    walk(node)
```

## 6. Dimension 方向转负：必须作为标准字段进入结果表

你认可“负 IC 通过 Dimension 方向转负”这个方案，我也建议二阶段必须贯彻。当前 `metrics.py` 的 `score` 逻辑是：先算 train/test 的 mean\_rank\_ic，如果 train 和 test 同号则 `same_sign = 1.0`，否则 0.25；最终 `score = abs(test_mean_rank_ic) * abs(test_rank_icir) * same_sign`。这能让稳定负 IC 因子排名靠前，但没有显式记录方向，也没有把因子值方向统一后再做后续统计。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/metrics.py "feature-init-factor-mining-427/src/alpha_mvp/metrics.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

改法：

```
def infer_dimension(train_rank_ic_series):
    med = np.nanmedian(train_rank_ic_series)
    if not np.isfinite(med) or abs(med) < 1e-6:
        return 0
    return 1 if med > 0 else -1

```

评价时：

```

metrics = summarize_factor_split(factor_oriented, fwd, dates, ...)
```

结果表新增：

```

oriented_train_mean_rank_ic double,
oriented_test_mean_rank_ic double,
oriented_rank_icir double,
oriented_quantile_spread double

```

最终保存：

```

dimension = -1
effective_expr = Neg(expr) 或 factor_value * -1

```

不建议真的生成 `Neg(expr)` 字符串参与 canonical；更好的方式是在结果表里记录 `dimension=-1`。这样同一公式不会因为方向不同变成两个表达式，统计也更干净。

## 8. 粗筛与细筛：Top-K + 概率抽样尾部

你提出粗筛后可以“选前多少名”或“按照统计分布抽样，越靠前概率越高，越到后面概率越低”。这个非常适合二阶段，因为模板扩展后表达式数量可能从 9000 上升到几万甚至几十万，如果只拿 top-K，可能会错过某些分数暂时一般但结构有潜力的模板族；如果全量细筛，计算量又太大。

建议 `candidate_sampler.py`：

```
def sample_candidates(df, top_k=2000, sample_n=3000, alpha=0.85, min_per_family=50):
    df = df.sort_values("coarse_score", ascending=False).reset_index(drop=True)
    top = df.head(top_k)

    tail = df.iloc[top_k:].copy()
    rank = np.arange(1, len(tail) + 1)
    p = 1 / np.power(rank, alpha)
    p = p / p.sum()

    sampled = tail.sample(n=min(sample_n, len(tail)), weights=p, random_state=seed)

    family_keep = (
        df.groupby("template_family", group_keys=False)
          .apply(lambda x: x.nlargest(min_per_family, "coarse_score"))
    )

    return concat_unique([top, sampled, family_keep])

```

候选选择逻辑：

```
coarse_candidates =
  top_k_by_score
  + probabilistic_tail_sample
  + template_family_min_quota
  + field_operator_promising_quota

```

这样既保留强者，也保留探索性。

## 12. 性能设计：先优化模板与去重，再优化计算

当前 `evaluator.py` 对 `Rank` 调用 `fastops.fast_rank_cs`，对 `TsCorr` 调用 `fastops.fast_rolling_corr`，其他 rolling 算子走 `ops.ROLLING`；`fastops.py` 已经用 numba 实现了横截面 rank、daily corr、rolling corr。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/evaluator.py "feature-init-factor-mining-427/src/alpha_mvp/evaluator.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

二阶段性能优化顺序建议：

### 12.1 第一优先级：减少重复表达式

用 canonical 去重只是第一层，还要做：

```
same_expr_hash skip
same_run_signature result skip
same_feature_set_hash + same_eval_hash + same_expr_hash reuse
same_template_family budget cap

```

### 12.2 第二优先级：预计算 primitive

比如对所有字段、所有窗口预计算：

```
TsMean($f,w)
TsStd($f,w)
TsDelta($f,w)
TsEMA($f,w)
TsWMA($f,w)
Rank(TsMean($f,w))
Rank(TsDelta($f,w))

```

存入 evaluator cache 或 DuckDB 临时目录。这样二阶、三阶模板只是在 primitive 上做组合，速度会明显提升。

### 12.3 第三优先级：扩展 fastops

优先将这些 pandas rolling 改成 numba 或 bottleneck：

```
TsMean
TsStd
TsDelta
TsDiv
TsEMA
TsWMA
TsRank

```

当前 `ops.py` 的 rolling 多数基于 `pd.DataFrame(x).rolling(...)`，在表达式数量扩大后会成为瓶颈。([GitHub](https://github.com/789453/feature-init-factor-mining-427/blob/fast-update-427/src/alpha_mvp/ops.py "feature-init-factor-mining-427/src/alpha_mvp/ops.py at fast-update-427 · 789453/feature-init-factor-mining-427 · GitHub"))

### 12.4 第四优先级：分 shard 跑

表达式任务天然可分片：

```
--shard-id 0 --num-shards 8
--shard-id 1 --num-shards 8
...

```

DuckDB job table 记录 shard 状态，最后统一 merge。

## 15. 最终二阶段范式

最后，我建议把二阶段定义成下面这套闭环：

```
选择字段
→ 统一模板生成
→ AST 合法性校验
→ canonical + signature 去重
→ DuckDB job queue
→ coarse pool 快速筛选
→ Dimension 方向统一
→ Rank 后加权 score
→ Top-K + 概率尾部抽样
→ fine pool 复核
→ 字段/算子/窗口/模板族 attribution
→ 根据统计结果更新 TemplateSpec

```

这个闭环正好对应你的两个核心提升：

第一，**通过更完善、更丰富、更灵活的模板来提升**。这不是简单加几个 `TsZScore` 或 `Rank`，而是建立一套统一模板文法，让一阶、二阶、三阶、四阶、多窗口、多算子组合都能受控生成。

第二，**通过统计反馈判断字段和模板逻辑是否有效**。每个表达式都要知道自己用了哪些字段、算子、窗口、模板族、复杂度；每次挖掘结束后，不只是看 top100，而是看字段 uplift、算子 uplift、模板族 survival、窗口分布、交互组合表现、样本外稳定性。

这套方案比“直接上 GP / LLM / PySR”更贴近你现在的目标：不复杂化，不学院化，保留单模块可运行特性，但把挖掘范式做完整。真正重要的是：**每次挖掘不是一次性跑结果，而是积累模板系统的证据。**
